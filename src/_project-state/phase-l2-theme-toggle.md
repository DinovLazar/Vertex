## What was built
`ThemeToggle` client component renders a sun/moon icon button that flips the site's theme via `useTheme()` from Phase L1's `ThemeProvider`. Wired into the Navbar in two places: immediately left of the desktop language toggle in the right cluster, and inside a new top-right cluster in the mobile hamburger overlay (alongside the language toggle, which was relocated from its previous position at the bottom of the overlay's flex-col to keep the two toggles visually paired across viewports). Uses `lucide-react` `Sun` / `Moon` icons with an `AnimatePresence`-driven crossfade (opacity + rotate + scale) on swap. Translated aria-labels added to both `messages/en.json` and `messages/mk.json` under `nav.themeToggle.{toLight,toDark}`. SSR-safe mount gate prevents icon flicker on first paint. Session C `focus-ring` utility applied for keyboard focus visibility. 44├Ś44 touch target met on both viewports via the hand-tuned className that matches the language toggle's pill shape.

## Adaptations from the plan's assumptions
The plan was written against a reasonable-but-optimistic guess at the language toggle's structure. Two adaptations:

1. **The language toggle does NOT use any CVA `icon-*` size.** It uses `<Button variant="ghost">` with `size="default"` (implicit), but completely overrides the default size via className: `min-h-[44px] gap-1.5 h-auto px-3 py-1.5 rounded-full`. The plan proposed a `buttonSize: 'icon' | 'icon-sm' | 'icon-xs' | 'icon-lg'` prop on the ThemeToggle with default `icon-sm` (28├Ś28) ŌĆö none of those match the 44px-tall pill shape. Dropped the `buttonSize` prop entirely and instead hand-tuned a `BUTTON_SHAPE` constant that mirrors the language toggle's className recipe, with the addition of `min-w-[44px]` since an icon-only button has no text to give it width. Caller passes `className` for responsive visibility (`hidden md:inline-flex` on desktop; undefined for mobile).

2. **The plan's nominal mount-gate approach ŌĆö render a `pointer-events-none opacity-0` placeholder before mount, swap to the real button after ŌĆö triggered a stuck CSS transition in this codebase.** The Button primitive (`@base-ui/react`'s `Button` via shadcn cva) includes `transition-all` in its base classes. When React patched the className from "has opacity-0" to "no opacity-0" on mount, the browser started a 150ms opacity transition but registered it with `currentTime: 0` and never advanced ŌĆö leaving the toggle permanently at `opacity: 0` despite `fill: backwards` and running playState. Reproduced reliably across fresh reloads; `getAnimations()` showed two animations (opacity + color) both stuck at `currentTime: 0, playState: "running"`; cancelling them manually via `anim.cancel()` instantly set opacity to 1. Rewrote the component so the **outer `<Button>` className stays STABLE across the mount boundary** ŌĆö only `aria-label`, `aria-hidden`, `onClick`, `tabIndex`, `title` toggle, plus the inner content swaps from an empty placeholder `<span>` to the real `AnimatePresence`-wrapped icon. With no className diff, no transition is registered, no transition can get stuck. SSR still renders an icon-free placeholder (verified via `curl -s /en | grep <button>` ŌĆö server output has `aria-hidden="true"` with an empty span inside, no Sun or Moon SVG), so the no-wrong-icon-flash guarantee is preserved.

## Key technical decisions
- **Stable outer / swappable inner, NOT opacity-0-placeholder.** The plan's mount-gate design used `pointer-events-none opacity-0` on a placeholder `<Button>` pre-mount, and removed those classes post-mount. This is a standard no-flicker pattern for manually-styled buttons, but it interacts poorly with the shadcn Button primitive's base `transition-all` class: React's className patch post-mount triggers a browser CSS transition on `opacity`, and that transition gets registered with `currentTime: 0, playState: "running"` and never advances ŌĆö verified via `getAnimations()` in the dev preview. The fix: keep the outer `<Button>` className STABLE across mount (same classes SSR and post-mount), and swap only the attributes that must change (`aria-label`, `aria-hidden`, `tabIndex`, `onClick`, `title`) + the inner content. React patches attributes without triggering `transition-all` opacity changes. SSR still renders the icon-free placeholder (empty inner `<span>`), so the no-wrong-icon flash guarantee is preserved.
- **Icon semantics: show the TARGET state, not the current state.** Currently dark ŌåÆ show Sun (clicking goes to light); currently light ŌåÆ show Moon. This is the web's dominant convention and matches user expectation that the icon represents the action's destination, not where you are now.
- **`AnimatePresence mode="wait"` + `motion.span`-inside-Button.** `mode="wait"` ensures the Sun exit animation completes before the Moon enter animation begins ŌĆö prevents both icons being briefly visible at the same time during the crossfade. The motion-animated element is the inner `<span>`, not the outer `<Button>` ŌĆö animating the Button would also animate its hover/focus treatment, which looks glitchy during the swap.
- **`buttonSize` prop dropped.** The plan proposed `buttonSize: 'icon' | 'icon-sm' | 'icon-xs' | 'icon-lg'` with default `icon-sm` (28├Ś28) so callers could match the language toggle. Reality: the language toggle does not use any of those sizes ŌĆö it uses `size="default"` (implicit, 32├Ś32) with complete className override to a 44px pill shape. None of the CVA icon sizes match. Replaced the prop with a `BUTTON_SHAPE` string constant that mirrors the language toggle's className recipe (`min-h-[44px] min-w-[44px] h-auto w-auto px-3 py-1.5 rounded-full hover:bg-white/5`), plus a `className` prop pass-through for responsive visibility and other per-callsite overrides. Simpler API, better match.
- **Desktop placement immediately left of the language toggle.** Matches the user's approved plan. Clusters the two preference controls (theme + language) visually, separated from the contact CTA.
- **Mobile overlay: language toggle relocated to top-right cluster.** Previously (Phase 6 / Session C) the mobile overlay placed the language toggle at the bottom of its flex-col, inside a `<motion.div className="mt-2">` after the "Get in Touch" CTA. Per the user's approved plan, the theme + language toggles are now paired in an `absolute top-[72px] right-4 flex items-center gap-2` cluster at the top of the overlay. The `top-[72px]` y-offset is deliberate: the fixed header is `h-14` (56px) on mobile and stays at z-50 above the overlay at z-40. The X close button lives in that header, anchored top-right. Placing the new cluster at y=72px puts it 16px below the header's bottom edge, so it never visually collides with the X control. (The inner container's pre-existing `pt-24` / 96px top padding means the centered nav links still start at y=96px, below the new cluster.)
- **Theme toggle inside overlay does NOT close the overlay.** Users can see the theme flip reflected on the page behind the still-open menu before choosing to dismiss it. Achieved by default: `ThemeToggle` doesn't accept a close-menu handler. Language toggle retains its `setMobileOpen(false); toggleLocale()` behavior from Phase 15A.
- **Import via relative path (`./ThemeToggle`), not the barrel.** The Navbar itself is re-exported from `@/components/global`. Importing `ThemeToggle` from the same barrel inside Navbar would be a minor circular-import risk; a direct relative import sidesteps that entirely.
- **`pointer-events-none` removed from the pre-mount state.** In the revised design, the pre-mount button's only functional differences from the post-mount button are `aria-hidden="true"`, `tabIndex="-1"`, no `onClick`, and no `aria-label`. The element IS interactively inert (clicks don't do anything because there's no handler; keyboard users can't tab into it; screen readers skip it). We don't need `pointer-events-none` on top of that. This simplifies the className diff to zero.

## What the next phase (L3) should know
- The toggle is live and functional. In light mode, the Navbar's `.glass-nav` backdrop, BorderGlow cards, and the three hero backgrounds (Silk on homepage, GridMotion panels on consulting, Plasma on marketing) still use dark-only baselines and visibly break. L3 fixes the heroes. L4 fixes the global chrome (`.glass`, BorderGlow, `.glow-hover`, `.metallic-panel`, chat-trigger pulse). L5 sweeps remaining sections and pages.
- The theme preference is observable from JavaScript at runtime via `useTheme()` in any client component inside `ThemeProvider`, or by reading `document.documentElement.dataset.theme` from any client-side context. L3 hero components may need to read the current theme to swap shader colors (Silk in particular ŌĆö the color is baked into a shader uniform and needs to update when the theme flips).
- The `ThemeProvider` emits state changes synchronously on `toggleTheme()`. React re-renders any consumer of `useTheme()` on the next render tick. If a shader component doesn't consume the hook, it won't react to theme changes ŌĆö L3 will need to either pass the theme down as a prop driven by a parent `useTheme()`, or have the shader read `data-theme` from an effect triggered by a `MutationObserver` on `<html>`.
- The Button primitive's `transition-all` + opacity-toggle interaction noted above (stuck CSS transition at `currentTime: 0`) is a general gotcha in this codebase. Any future client component that mounts a Button whose className changes from "hidden/invisible" to "visible" on mount should use the same stable-outer / swappable-inner pattern, not an `opacity-0` class toggle.
