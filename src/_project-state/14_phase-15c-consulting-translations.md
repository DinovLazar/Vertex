# Phase 15C — Consulting Division Translations

## What was built
Every string on the consulting landing + 4 consulting service pages (`/consulting`, `/consulting/business-consulting`, `/consulting/workflow-restructuring`, `/consulting/it-systems`, `/consulting/ai-consulting`) now reads through `getTranslations` (server) or `useTranslations` (client). `/mk/consulting/*` pages render fully in Macedonian — hero, body prose (~800 words each, 5 sections per page), bullet lists (including the `<strong>` term+description AI bullets), process steps, FAQ items (with FAQPage JSON-LD in MK), related-services chips, and CTA banner.

`ConsultingServicePage.tsx` was restructured away from JSX-as-prop to a structured data prop (`content: ContentSection[]`), making service page bodies 100% translation-driven. `ConsultingServicesGrid.tsx` is now prop-driven and reusable. Each service page is a server component that awaits `params`, calls `setRequestLocale`, and feeds the template from `consulting.<service>.*` translation keys plus the shared `consulting.serviceCommon.*` chrome labels.

## Key technical decisions
- **`content` is structured data, not JSX.** The old `ConsultingServicePage.tsx` took `content: React.ReactNode` and every caller passed a `<>...<h2>...</h2><p>...</p>...</>` fragment. That made translations impossible without encoding JSX in JSON. Swapped to `ContentSection[]` with optional bullets + optional post-bullet paragraphs. The template maps over it. This is strictly more capable — it supports everything the old JSX did for consulting pages (5 headings + paragraphs + bullets + post-bullet paragraphs) except raw HTML like inline `<strong>` inside paragraphs, which no consulting page actually used. For bullets specifically, the `Bullet.term?` field handles the AI Consulting `<strong>Document processing</strong> — ...` pattern.
- **Service page files are server components.** Next 16 async `params` works cleanly. `setRequestLocale(locale)` at the top of the render ensures both `getTranslations` (called directly) and `useTranslations` (used by descendant client components like `ProcessSteps`, `FAQAccordion`, `CTABanner`) see the right locale during static generation. Without `setRequestLocale`, next-intl would warn / fall back at SSG time.
- **`consulting.serviceCommon.*` holds the shared chrome strings.** Overline ("Vertex Consulting"), process section title ("Our process"), FAQ section title ("Frequently asked questions"), related section title ("Related services"), and the default CTA banner copy (formerly hardcoded in `ConsultingServicePage.tsx`) all live there. Each service page pulls from this namespace via a second `getTranslations` call. Cleaner than duplicating per-service and means the "Contact Us" button wording is locked across all four service pages.
- **Related-services titles reuse `nav.dropdown.*` values.** Rather than duplicating "Workflow Restructuring" / "Преструктуирање на процеси" into every service's `related.links[].title`, I stored the **title directly in each service's related array**. This is the plan's recommended approach and avoids runtime indirection. Discipline: if Goran renames a service, he needs to update `nav.dropdown.*` + each page's hero title + each peer page's related links — I flagged this in `TRANSLATION_NOTES.md` convention #13 and the 15C callout. Four locations per service, caught by a full-text search.
- **`consulting.landing.services.*.description` is distinct from `home.servicesOverview.services.*.description`.** The homepage services grid and the consulting landing services grid have **different** English copy — both read authoritative strings from their own dictionaries, only the titles (from `nav.dropdown.*`) are shared. Kept that way: homepage has a marketing hook, landing has a "here's how we help" framing.
- **`ConsultingServicesGrid` stayed client.** It uses `motion.div` + `staggerContainer` / `staggerItem` variants. Making it prop-driven doesn't change that. The consulting landing builds the array client-side in `ConsultingLandingClient` (also `'use client'`), which is fine because client components can call `useTranslations` directly under `NextIntlClientProvider`.
- **Not translating per-page `<title>` templates.** Each consulting service page's title renders as "{MK title} | Vertex Consulting" because `src/app/layout.tsx` sets `title.template = "%s | Vertex Consulting"` once globally. The "Vertex Consulting" suffix stays in Latin on `/mk` — intentional, it's the brand wordmark. If Goran wants "| Vertex Консалтинг" or similar on `/mk`, that's a separate decision.
