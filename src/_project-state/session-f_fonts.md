# Session F — Font System Overhaul

## What was built

Swapped the two loaded Google fonts: **Sora → Manrope** (headings) and **DM Sans → Onest** (body). Both new fonts load the `cyrillic` and `cyrillic-ext` subsets in addition to `latin` and `latin-ext`, which resolves the Phase 15F-flagged blocker (Macedonian content previously rendered in OS fallback fonts because neither Sora nor DM Sans had Cyrillic coverage via `next/font/google`). The swap also moves the codebase off DM Sans, which is on the impeccable skill's reflex-default typography reject list — both issues solved in one pass. Also renamed the underlying CSS variable names from `--font-sora` / `--font-dm-sans` to the semantically correct `--font-heading` / `--font-body` so the codebase no longer keeps references to font names that don't match reality.

## Key technical decisions

- **Manrope + Onest over alternatives** (Manrope + Golos Text, Manrope + Inter, Manrope + IBM Plex Sans). Chose this pair for three reasons: (1) both faces publish `cyrillic` and `cyrillic-ext` subsets on Google Fonts with genuinely well-drawn Cyrillic letterforms, so Macedonian content renders with the same stroke weight / x-height / geometric feel as the Latin copy on the EN side; (2) Onest is meaningfully less common than DM Sans (which helps distinctiveness), while staying geometric-humanist enough that the pairing with Manrope reads as one cohesive voice across display and body sizes; (3) neither appears on the impeccable skill's reflex-default reject list.
- **CSS variable rename** (`--font-sora` → `--font-heading` and `--font-dm-sans` → `--font-body`). Done in the same pass as the swap. Keeping the old variable names with new font values was considered but rejected — leaves future maintainers confused when `--font-sora` resolves to Manrope, and the whole reason this rename was viable was that we're already touching every usage site anyway. Every `.prose-*` rule and inline `fontFamily` string was updated to reference `var(--font-heading)` instead of `var(--font-sora)`.
- **`@theme inline` for font tokens — Option A over Option B.** Renaming the next/font variables to `--font-heading` / `--font-body` collides with the existing Tailwind-v4 token names. Two patterns could resolve it: Option A (move the font tokens into a `@theme inline` block so Tailwind skips the `:root` emission and uses `var(--font-heading)` directly as the utility's `font-family` value, with next/font's class on `<html>` as the sole source) or Option B (rename the Tailwind tokens to something like `--font-sans-heading` / `--font-sans-body` and rewrite every `font-heading` / `font-body` utility class across ~30 component files). Option A chosen because (i) `globals.css` already uses `@theme inline` for the shadcn tokens at lines 142+, so it's not introducing a new pattern — just extending an existing one, (ii) zero component-level changes needed, (iii) the `system-ui, sans-serif` tail is preserved inside the utility declaration itself so even if next/font somehow fails to inject, there's a graceful browser fallback.
- **Weight arrays:** Manrope loaded at 400/500/600/700/800 — covers body default through 600-weight semibold subheads, 700-weight bold headlines, and 800-weight display. Onest loaded at 400/500/600/700 — body default through bold emphasis runs. Weight 500+ shows as "loaded" for all four subsets in `document.fonts` on a typical page visit; 400 and 800 lazy-load as actual uses appear on the page. Any unused weight can be trimmed in a future pass for marginal bundle savings.
- **Cyrillic subset loading:** both fonts include `cyrillic` (U+301, U+400-45F, U+490-491, U+4B0-4B1, U+2116) and `cyrillic-ext` (U+460-52F, U+1C80-1C8A, U+20B4, U+2DE0-2DFF, U+A640-A69F, U+FE2E-FE2F) for full Macedonian coverage. Verified by enumerating `document.fonts` on a `/mk/consulting/ai-consulting` load — both ranges show `status: "loaded"` for weights 500/600/700 (the weights actively in use on that page).

## Visual regression findings

- **EN/MK parity clean across 16 routes.** Walked homepage + consulting landing + 4 consulting service pages + marketing landing + 4 marketing service pages + about + contact + blog listing + a blog post + privacy + thank-you on both `/en/...` and `/mk/...` via the dev server. Cyrillic H1s (e.g. `Помагаме на бизнисите да растат попаметно.` on `/mk`, `AI консалтинг` on `/mk/consulting/ai-consulting`) render in Manrope, not the OS fallback — stroke weight, x-height, and terminals match the Latin copy.
- **Prose h2/h3 migration worked (after a dev-server restart).** The 6 `.prose-*` h2/h3 rules were updated from `var(--font-sora)` → `var(--font-heading)` in the source, but Turbopack's CSS bundle was caching the pre-edit CSS chunk — the compiled stylesheet kept serving `font-family: var(--font-sora)` even after HMR reported "updated". Dev server was stopped, `.next/` cache wiped, preview restarted; post-restart the stylesheet serves the correct `var(--font-heading)` rule and `.prose-consulting h2` / `.prose-marketing h2` / `.prose-blog h2` all resolve to `Manrope, "Manrope Fallback"` at runtime. Worth flagging: this was the same class of Turbopack-caches-stale-artifacts bug that `current-state.md` already documents for `src/proxy.ts`, just this time the cached artifact was a hand-written global CSS rule rather than the edge proxy handler.
- **Mobile heading overflow clean on MK.** Viewport resized to the mobile preset (375×812, effective content width 343px after scrollbar reserve) and the three landing-hero H1s on `/mk`, `/mk/consulting`, `/mk/marketing` all fit inside the container without clipping — `scrollWidth === clientWidth` for each. The `overflow-wrap: anywhere` safety-net on `h1..h6` (inherited from Finishing-Touches-v2 Phase 4 Bug 1) is still in place in `globals.css` line 301 and was not modified. Cyrillic character widths are ~5–10% wider per-glyph than Latin equivalents, but the fluid `text-display` clamp (`clamp(36px, 24.4px + 3.1vw, 64px)`) scales down far enough on mobile (to ~36px on a 375px viewport) that the wider Cyrillic glyphs still wrap cleanly into 3 lines on the shorter headlines and 4 lines on `/mk/marketing`'s longer headline.
- **FOUT observation.** Hard-reloaded 4 different pages (homepage EN, homepage MK, a consulting service page MK, a blog post MK). The initial paint either shows no fallback flash or a very brief one before Manrope + Onest take over — well under the 200ms ceiling the plan specified. `display: 'swap'` is configured on both fonts, so the browser renders with the next/font-registered `Manrope Fallback` / `Onest Fallback` metric-matched fallback fonts until the actual Manrope/Onest woff2 files finish downloading, then swaps in. The metric-matched fallbacks are the reason the transition isn't perceived as a size/layout jump — they're sized to produce identical layout to the real font.

## Known issues

None directly from Session F. Two observations worth noting for future maintainers:

- **Turbopack CSS caching gotcha.** When editing hand-written rules inside `globals.css` (as opposed to component `className` strings that Tailwind JIT compiles fresh), HMR sometimes reports "updated" without actually rebuilding the compiled CSS chunk. The symptom is that the dev server keeps serving the pre-edit CSS even though the source file on disk is correct and the browser has reloaded. Fix: stop the preview server, `rm -rf .next`, restart. Session F hit this on the `.prose-*` rule changes; `current-state.md` already documents the same pattern for `src/proxy.ts` edits.
- **Manrope weight 800 is loaded but not yet used.** Included in the weight array for future display-size headings (audit flagged typography hierarchy as a possible polish target). Currently no heading utility sets `font-weight: 800`, so weight 800's font files will load only if a future caller opts in. Can be trimmed from the weight array without visual impact if bundle size becomes a concern.

- **Font system naming conventions.** The next/font-injected CSS variables (`--font-heading` / `--font-body`) now share namespace with the Tailwind-v4 token names. If a future session adds a third font (e.g. a monospace face for code snippets), follow the same pattern: pick a semantic name like `--font-mono`, declare it in the `@theme inline` block at the top of `globals.css` (not the plain `@theme` block), and set `variable: '--font-mono'` on the next/font call. Don't reintroduce font-specific variable names like `--font-inter` — they rot the moment the font is swapped.
- **`@theme inline` pattern is load-bearing.** The Session F setup relies on Tailwind v4's `@theme inline` skipping `:root` emission for the font tokens. If a future Tailwind upgrade changes this behavior, the fallback-font tail (`system-ui, sans-serif`) could silently drop from the computed font-family — worth smoke-testing `getComputedStyle(h1).fontFamily` after any Tailwind major-version bump.
