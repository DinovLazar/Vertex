## Key technical decisions

- **CTABanner background token choice: `var(--color-ink)`.** `#0E0E0E` is an exact match for the `--color-ink` token defined in the `@theme` block (`src/app/globals.css:20`). The Finishing-Touches-v2 intent for the CTA banner was to sit on the deepest dark surface ŌĆö deeper than `--division-bg` (`#141414`) which is the page background. `--color-ink` is that deepest surface. Exact hex match, exact semantic match. Verified at runtime: `getComputedStyle(ctaSection).backgroundColor === 'rgb(14, 14, 14)'`.
- **DivisionSplit: Option A over Option B.** Option B (keep the fields, swap hex for `var(--division-accent)` / `var(--division-card)` references) was the safer path per the plan but doesn't work here ŌĆö the component uses the `${division.accentColor}30` alpha-suffix pattern in 4 places (hover border, icon bg, service-tag bg, service-tag text color), which requires a concrete hex string. CSS variables can't be suffixed with 2-char hex alpha values. Option A (delete the fields entirely, inline tokens + rgba values) was cleaner: no dead per-division fields in the data array, no hex in the source, and the alpha overlays now use explicit `rgba(245, 245, 245, ╬▒)` with opacity values hand-derived from the original hex alphas (30 hex ŌåÆ 0.19, 15 hex ŌåÆ 0.08, 10 hex ŌåÆ 0.06, CC hex ŌåÆ 0.8). Doc comment updated in the DIVISION_KEYS declaration to explain.
- **`gradient-text*` migration target: inline `style` instead of a `text-[var(--division-text-primary)]` utility class.** The original plan called for class-based migration: `className="gradient-text"` ŌåÆ a class setting the primary-text color via Tailwind arbitrary value. But `ConsultingLandingClient.tsx` carried a long warning comment from a previous failed attempt: passing that utility as `headlineClassName` caused tailwind-merge to classify it as a text-size utility and strip the base `text-display` class, collapsing the headline from a fluid 40ŌĆō80px display to the inherited 16px body size. (Arbitrary-value text utilities are ambiguous between color and size; tailwind-merge can't disambiguate without a type hint prefix like `color:` inside the brackets.) Rather than add type-hint syntax to every migration site, the fix routes color via inline `style` ŌĆö which can't be stripped by tailwind-merge and doesn't collide with any utility class. Runtime verification on all 3 landing pages: h1 `fontSize === '40.024px'` (correct `text-display` at the test viewport) + `color === 'rgb(245, 245, 245)'`. The stale warning comment in `ConsultingLandingClient.tsx` was removed since the inline-style approach makes the tailwind-merge caveat moot.
- **HeroSection `headlineClassName` prop kept, default dropped.** The prop stays in the public API for forward-compat ŌĆö a future caller might want to override the className (e.g., add `italic` or `tracking-tighter`). But the default value is now `undefined`, not `'gradient-text-brand'`, because none of the 3 existing callers override it anymore post-grayscale. The base color is always applied via inline style regardless of what the caller passes.
- **BorderGlow border removal confirmed safe.** The `::before` pseudo-element (colored mesh-gradient border), `::after` pseudo-element (colored mesh-gradient fill), and `.edge-light` child element (outer glow) are all positioned `inset: 0` / `inset: calc(var(--glow-padding) * -1)` with `z-index: -1` or pointer-events:none, and they all opacity-ramp from 0 to ~proximity when the pointer is near the edge. Dropping the static `border: 1px solid rgb(255 255 255 / 15%)` only affects the rest state ŌĆö on hover, the glow layers already carried the visual edge treatment. The card's `padding-box` / `border-box` geometry is preserved by the implicit `0px` border fallback. Verified at runtime: `getComputedStyle(card).borderWidth === '0px'` on 4 consulting service cards and 8 homepage services cards; hover glow still fires on pointer-move.
- **Hardcoded hex exceptions preserved (documented in grep sweep results).** Five categories of hex values remain in source files and are intentionally left untouched: (1) token definitions inside `@theme` and `[data-division="..."]` blocks in `globals.css` ŌĆö those ARE the tokens, (2) `BorderGlow colors={[...]}` props in 6 components ŌĆö the pointer-follow gradient math uses radial-gradient interpolation that needs concrete hex, (3) WebGL shader color props on `BackgroundSilk`/`Plasma`/`Grid` ŌĆö GPU shaders don't resolve CSS variables, (4) `themeColor: '#0E0E0E'` in root layout metadata ŌĆö mobile browser chrome color, must be a literal hex in the `<meta>` tag, (5) email HTML in `/api/contact/route.ts` + `/api/newsletter/route.ts` ŌĆö email clients can't resolve CSS variables. Also flagged: `src/lib/divisions.ts:15-29` still carries `text-[#F5F5F5]` / `bg-[#141414]` / `border-[#404040]` Tailwind arbitrary-value classes; that file is the canonical Tailwind-class bridge for the division palette and is semantically part of the token system, so leaving untouched is the right call.
