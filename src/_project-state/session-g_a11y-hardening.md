## Key technical decisions

- **Root layout as pass-through, not "locale-neutral html + body".** Phase 15A had left `<html>` on the root layout without a `lang` attribute and moved the `<div lang={locale}>` into the locale layout to avoid hydration mismatches. That pattern works for screen-reader correctness at the paragraph level but fails WCAG 3.1.1 which evaluates `document.documentElement.getAttribute('lang')`. The next-intl recommended pattern is to let the locale layout own the `<html>` + `<body>` entirely and turn the root layout into a pass-through — Next.js accepts this because the locale layout provides the HTML shell. Metadata and viewport stay exported from the root layout (Next.js merges metadata across the whole segment tree). `/api/*` routes never invoke layouts so they're unaffected by the move.
- **Hybrid Link + disclosure-button pattern for Navbar dropdowns.** Considered three options: (a) keep the single `<Link>` and put the disclosure ARIA on the link — rejected because a link opening a disclosure is semantically wrong (links navigate; buttons perform actions); (b) replace the Link with a button, losing the parent-page navigation — rejected because `/consulting` and `/marketing` are real destinations; (c) split into Link + separate disclosure button — picked. The Link preserves navigation + the `layoutId="nav-underline"` active underline; the adjacent button owns the disclosure semantics. Visually the two sit in one `flex.items-center` container and read as a single "Consulting ▾" unit. The `layoutId` underline moved up to the wrapper so it still spans the visible element.
- **`aria-haspopup="menu"` but no `role="menu"` on the children.** The user's plan specified `aria-haspopup="menu"` AND "Tab traverses children". These are in mild tension — strict WAI-ARIA menu pattern uses arrow-key navigation and `role="menuitem"` children with `tabindex="-1"`. The compromise (common in real-world nav dropdowns): use `aria-haspopup="menu"` as a type-hint for screen readers, but keep the children as plain `<Link>` elements without `role="menuitem"` so Tab-traversal works as expected. This matches the W3C ARIA Authoring Practices "Disclosure Navigation" example, which uses plain links inside the disclosed region.
- **Focus trap via `inert` on background, not via a focus-trap library.** React 19 natively supports `inert` as a boolean attribute, and modern browsers expose `element.inert` as an HTMLElement property. Toggling `main.inert = true` + `footer.inert = true` when the mobile menu opens effectively creates a focus trap — Tab can only reach elements inside the overlay and the navbar header, and the browser handles the skip-over logic for us. No dependency added. The `<header>` stays interactive so the hamburger's X button (and the logo, which navigates away from the trap) remain reachable.
- **Imperative `inert` toggling (useEffect + document.querySelector) instead of React-managed.** The `<main>` and `<footer>` live in `src/app/[locale]/(site)/layout.tsx` — sibling components to the `<Navbar />`. Sharing the mobile-open state via Context would have required converting the site layout to a client component (loses server-component benefits — `getTranslations` for the skip link, etc.) and adding a new Provider. Imperative DOM manipulation from the Navbar's useEffect is a cleaner fit: the Navbar already owns the state, it's the only component that needs to touch siblings, and `main` / `footer` are stable selectors. Cleanup in the effect's return function ensures inert is always cleared if the component unmounts with menu open.
- **Field-level errors with a retained banner for API failures.** The plan called for `aria-invalid` + `aria-describedby` + `role="alert"`. The simplest implementation was to keep the single bottom banner and point `aria-describedby` at it from the failing field. That works but is awkward — one banner with text like "Please enter your name" associated with the email input reads oddly. The better implementation, which the plan hints at with "matching ids", is field-level errors: each of name/email/message gets its own inline error rendered below itself with a unique id. The existing bottom banner is kept for submission-level errors (API / network failures after validation passes). On validation failure, focus moves to the first invalid field — so screen readers announce the associated error immediately.
- **`useId()` everywhere instead of static ids.** Every id in this session comes from React's `useId()` hook: `${uniqueId}-name-error`, `${uniqueId}-menu-nav-consulting`, `${uniqueId}-panel-0`, etc. This is slightly more verbose than `"name-error"` / `"faq-panel-0"` but guarantees uniqueness when two instances of the same component render on the same page (e.g., two FAQAccordions on a service page — the process-FAQ pattern could do this in the future). Stable across SSR/CSR, no collisions, no need to thread a prop.
- **Skip-link styled with inline `style` for tokens, Tailwind for layout.** The skip-link visibility transition uses Tailwind utilities (`fixed left-4 top-4 -translate-y-[200%] focus-visible:translate-y-0 transition-transform px-4 py-2 rounded-button`) but the colors (background, foreground, outline) use inline `style={{ backgroundColor: 'var(--division-accent)', color: 'var(--division-bg)', outlineColor: 'var(--division-bg)' }}`. This mirrors Session E's HeroSection pattern — inline style wins over tailwind-merge arbitrary-value class collisions and keeps the component self-contained (no new CSS utility class needed in `globals.css`). No hardcoded hex.
- **`:focus-visible` on the skip-link instead of `:focus`.** A sighted mouse user who clicks the skip link (unlikely but possible) gets the default focus behavior; the link becomes visible only on keyboard-driven focus, which is the intended use case. Programmatic `.focus()` in some browsers doesn't trigger `:focus-visible` — during Electron-preview testing we used `el.focus({ focusVisible: true })` to verify the computed styles, and real-keyboard Tab behavior matches this.

## Known issues

- **Motion entrance animations on already-aria-hidden content.** The mobile menu `AnimatePresence`-managed motion.div briefly animates opacity 1→0 during close. During that ~300ms window, `main.inert` has already flipped back to `false` (we clear it on the `mobileOpen === false` branch). If a user somehow triggers Tab during the exit animation, they could in theory tab into the still-fading overlay. In practice Escape/X-click returns focus to the hamburger before this matters, and the window is too short to exploit. If a future session wants to eliminate this edge case entirely, switch the overlay from `AnimatePresence` + conditional render to always-rendered with `inert={!mobileOpen}` — but that would require rethinking the exit stagger animation on the menu items.
- **`hasAttribute('aria-busy')` always returns true on React-controlled elements.** Even when `aria-busy={false}`, React renders the attribute as `aria-busy="false"` (not omitted). This is correct per HTML spec (the attribute is a token-valued boolean, not a presence boolean) but looks odd in devtools. Assistive tech treats `"false"` as "not busy" the same as omitted, so no behavior difference.
- **Mobile Safari and older versions of iOS may not fully honor `inert`.** Support landed in Safari 15.5 (May 2022). For users on iOS 14 or older, the focus trap would degrade — they could tab into the main content while the overlay is visible. Per caniuse this is <0.2% of global traffic in 2026. Not addressing this session; if it becomes a real issue, a lightweight focus-trap fallback via `focusin` listener could be added.

## What the next session should know

- **The root layout is now a pass-through.** Never re-add `<html>` / `<body>` / fonts / providers to `src/app/layout.tsx`. All of that belongs in `[locale]/layout.tsx`. The root layout stays minimal so API routes don't pay for layout tree evaluation and so the `<html lang={locale}>` attribute can be set correctly per request.
- **`<html lang={locale}>` is now the source of truth for document language.** If a future session adds a new locale (say `sq` for Albanian), update `src/i18n/routing.ts` `locales` array and the `lang` attribute flows automatically. The old `data-locale` wrapper div is gone.
- **Skip-to-content link pattern for any future full-page layouts.** If a new layout tree needs a skip link (e.g., a `/dashboard` route group), follow the `src/app/[locale]/(site)/layout.tsx` pattern: `async` server component, `getTranslations('common')`, render the `<a href="#main-content">` before the `<Navbar />`, give the `<main>` `id="main-content"`. Translation key `common.skipToContent` is reusable across layouts.
- **Navbar disclosure pattern is reusable.** If a future component needs a disclosure widget (e.g., a settings-menu dropdown, a user-profile flyout), follow the Navbar pattern: dedicated `<button type="button" aria-haspopup="menu" aria-expanded aria-controls aria-label>` trigger, `useId()`-derived `menuId`, `onKeyDown={Escape}` + `onBlur={focus-leaves-wrapper}` handlers at the wrapper level, `refs` map for focus return. The hover layer (`onMouseEnter` / `onMouseLeave` with `document.activeElement` check) is optional — only add it if the design calls for hover-to-open as progressive enhancement.
- **Form error pattern is reusable.** ContactForm is the canonical example. If a future form needs validation errors: track a `fieldErrors` object (not a single string), clear errors on `handleChange`, add `aria-invalid` + `aria-describedby` to each input, render inline `<p role="alert">` below the input, and focus the first invalid field on submit. Keep the submission-level banner (also `role="alert"`) for API failures.
- **`main` and `footer` are now inert-toggleable.** The imperative `document.querySelector('main').inert = true/false` pattern in the Navbar is the established convention. If a future modal / overlay needs the same focus-trap treatment, it can follow the same pattern from its own component. Avoid wrapping `<main>` in a client-component provider just to share the inert state — the imperative approach keeps the site layout a server component.
- **Translation keys added this session.** `common.skipToContent` (both locales), `nav.submenuToggleAria` (both locales, uses ICU `{label}` interpolation). Don't rename these — they're referenced from multiple places.
- **Audit dimensions now green.** After Session G the accessibility audit is clean across: document language (3.1.1), keyboard operability of menus (2.1.1 + 4.1.2), error identification + labels (3.3.1 + 3.3.3 + 4.1.2), focus management + skip nav (2.4.1 + 2.4.3), and decorative-icon exclusion (1.1.1 for icon-fonts). The only remaining non-Session-G a11y items would be contrast spot-checks on new surfaces + screen-reader-specific phrasing polish — both external / manual-review tasks, not code changes.
